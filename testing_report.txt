1. INTRODUCCIÓN
================================================================================

El microservicio "appointment-service" es el núcleo de gestión de citas
veterinarias del sistema PetWell. Permite a los dueños de mascotas reservar
citas, y a las clínicas administrarlas: confirmarlas, cancelarlas, editarlas
y marcarlas como completadas.

Este reporte documenta las pruebas implementadas para validar el correcto
funcionamiento del microservicio sin modificar el código fuente (src/).

Se implementaron:
  - 6 pruebas unitarias  (tests/unit/appointment.service.test.ts)
  - 6+ pruebas de integración  (tests/integration/appointment.test.ts)

================================================================================
2. PRUEBAS UNITARIAS (tests/unit/appointment.service.test.ts)
================================================================================

Las pruebas unitarias validan la lógica de negocio en el archivo:
  src/services/appointment.service.ts

Cada prueba aísla el servicio usando mocks, sin tocar la base de datos real.

----------------------------------------------------------------------
PRUEBA 1 — listAppointments (unit - 1)
----------------------------------------------------------------------
QUÉ VALIDA:
  - Que un DUENO_MASCOTA solo vea sus propias citas (filtradas por owner).
  - Que un CLINIC_ADMIN vea las citas de su clínica.
  - Que los errores de red en servicios externos (axios) no rompan el flujo
    (axios lanza error → se maneja silenciosamente).

RESULTADO ESPERADO:
  - Ambos roles retornan un arreglo con las citas correspondientes.

----------------------------------------------------------------------
PRUEBA 2 — getAppointment (unit - 2)
----------------------------------------------------------------------
QUÉ VALIDA:
  - Que un CLINIC_ADMIN pueda ver cualquier cita sin restricción de propiedad.
  - Que un DUENO_MASCOTA pueda ver su propia cita verificando owner_ids[].
  - Que también funcione con el campo "owners[].id" como alternativa.

RESULTADO ESPERADO:
  - La cita se retorna correctamente con datos enriquecidos.

----------------------------------------------------------------------
PRUEBA 3 — bookAppointment éxito TELEMEDICINA (unit - 3)
----------------------------------------------------------------------
QUÉ VALIDA:
  - Que se pueda reservar una cita de tipo TELEMEDICINA correctamente.
  - Que se llame al servicio de precios (billing) para obtener tarifa.
  - Que las notificaciones y sesión de telemed se disparen en background.

RESULTADO ESPERADO:
  - La cita se crea y retorna con id = 'appt_telemed'.

----------------------------------------------------------------------
PRUEBA 4 — bookAppointment caminos de error (unit - 4)
----------------------------------------------------------------------
QUÉ VALIDA:
  - Error 404 al verificar mascota (Mascota no encontrada).
  - Error 403 al verificar mascota (No autorizado para esta mascota).
  - Error genérico en verificación de mascota.
  - Que el dueño de otra mascota no pueda reservar (owner_id diferente).
  - Error 404 al verificar veterinario (Veterinario no encontrado).
  - Que un veterinario de otra clínica sea rechazado.
  - Error de comunicación con User Service (502).
  - Fallback de end_time cuando no hay bloques horarios definidos.

RESULTADO ESPERADO:
  - Cada caso lanza el error apropiado con mensaje descriptivo.
  - El caso de fallback crea la cita con duración por defecto de 30 min.

----------------------------------------------------------------------
PRUEBA 5 — editAppointment y changeAppointmentStatus (unit - 5)
----------------------------------------------------------------------
QUÉ VALIDA:
  - Edición exitosa de una cita como CLINIC_ADMIN.
  - Error al editar cita no encontrada (404).
  - Error de permisos al editar como DUENO_MASCOTA sin ser dueño (403).
  - Error al editar cita en estado CONFIRMED como DUENO_MASCOTA.
  - Error al editar cita de otra clínica como CLINIC_ADMIN.
  - Error al editar cita COMPLETED o CANCELLED.
  - Cambio de estado exitoso a CONFIRMED.
  - Error al cambiar estado de cita no encontrada.
  - Error de permisos como DUENO_MASCOTA sin ser dueño.
  - Que DUENO_MASCOTA solo pueda CANCELAR (no confirmar).
  - Que DUENO_MASCOTA solo cancele en estado PENDING/CONFIRMED.
  - Restricciones del rol VETERINARIO (solo COMPLETED/NO_SHOW).
  - Restricciones de clínica para CLINIC_ADMIN.
  - Que CANCELLED y COMPLETED disparen eventos correctamente.

RESULTADO ESPERADO:
  - Cada validación lanza el error correspondiente o ejecuta la acción.

----------------------------------------------------------------------
PRUEBA 6 — removeAppointment (unit - 6)
----------------------------------------------------------------------
QUÉ VALIDA:
  - Que se elimine correctamente una cita existente.
  - Error al intentar eliminar cita no encontrada (404).
  - Error de permisos al intentar eliminar cita de otra clínica (403).

RESULTADO ESPERADO:
  - Eliminación exitosa retorna undefined (void).
  - Los errores lanzan excepción con código y mensaje apropiados.

4. USO DE MOCKS
================================================================================

Un mock es un objeto falso que reemplaza una dependencia real durante los tests,
permitiendo que las pruebas sean rápidas, aisladas y no afecten datos reales.

----------------------------------------------------------------------
MOCK 1 — Repository (Base de datos / Supabase)
----------------------------------------------------------------------
MÓDULO MOCKEADO: src/repositories/appointment.repository.ts
                 src/repositories/vetblock.repository.ts

POR QUÉ:
  El repositorio hace consultas reales a Supabase. En los tests no queremos
  conectarnos a la base de datos real para no afectar datos de producción
  y para que los tests sean instantáneos.

FUNCIONES MOCKEADAS:
  - findAppointmentsByOwner   → simula lista de citas por dueño
  - findAppointmentsByClinic  → simula lista de citas por clínica
  - findAppointmentById       → simula búsqueda por ID
  - findByVetDateTime         → simula verificación de slot disponible
  - createAppointment         → simula creación de cita
  - updateAppointment         → simula actualización
  - patchAppointmentStatus    → simula cambio de estado
  - deleteAppointment         → simula eliminación
  - findVetBlocksByVetAndDay  → simula horarios del veterinario

CÓMO:
  jest.mock('../../src/repositories/appointment.repository');
  (appointmentRepo.findAppointmentById as jest.Mock).mockResolvedValue({ id: 'appt1', ... });

----------------------------------------------------------------------
MOCK 2 — Servicios externos (axios)
----------------------------------------------------------------------
MÓDULO MOCKEADO: axios

POR QUÉ:
  El servicio llama a Pet Service, User Service y Billing Service vía HTTP.
  Sin mock, los tests fallarían si esos microservicios no están corriendo.

ENDPOINTS SIMULADOS:
  - GET /api/v1/pets/:id          → datos de mascota (owner_id)
  - GET /api/v1/users/:id         → datos de veterinario (clinic_id)
  - GET /api/v1/billing/pricing/  → precios de la clínica
  - POST /api/v1/telemed/sessions → creación de sesión de telemedicina
  - POST /api/v1/notifications/   → envío de notificaciones

CÓMO:
  jest.mock('axios');
  mockedAxios.get.mockResolvedValue({ data: { data: { owner_id: 'u1' } } });

----------------------------------------------------------------------
MOCK 3 — Variables de entorno
----------------------------------------------------------------------
MÓDULO MOCKEADO: src/config/env

POR QUÉ:
  El módulo de configuración lee variables de entorno como SUPABASE_URL,
  JWT_SECRET, INTERNAL_SERVICE_KEY. En los tests las proveemos directamente
  para no depender del archivo .env.

CÓMO:
  jest.mock('../../src/config/env', () => ({
    env: {
      JWT_SECRET: 'test-secret',
      SUPABASE_SERVICE_ROLE_KEY: 'test-service-key',
      INTERNAL_SERVICE_KEY: 'test-service-key',
      ...
    }
  }));

5. USO DE MATCHERS (Verificadores de Jest)
================================================================================

Los matchers son funciones que comprueban que los valores resultantes son los
esperados. Jest provee muchos matchers; aquí los usados en este proyecto:

----------------------------------------------------------------------
toBe
----------------------------------------------------------------------
DESCRIPCIÓN: Verifica igualdad estricta (===). Ideal para primitivos.

EJEMPLO EN ESTE PROYECTO:
  expect(res.status).toBe(200);
  expect(res.body.success).toBe(true);
  expect(res.status).toBe(401);    ← verifica rechazo sin token

USO: Para verificar códigos HTTP, booleanos y strings exactos.

----------------------------------------------------------------------
toEqual
----------------------------------------------------------------------
DESCRIPCIÓN: Verifica igualdad profunda de objetos y arreglos.
             A diferencia de toBe, compara el contenido, no la referencia.

EJEMPLO EN ESTE PROYECTO:
  expect(res.body.data.status).toEqual('CONFIRMED');
  expect(res.body.data.reason).toEqual('Updated');
  expect(res.body.message).toEqual('Cita eliminada');

USO: Para verificar propiedades de respuestas HTTP y objetos retornados.

----------------------------------------------------------------------
toHaveProperty
----------------------------------------------------------------------
DESCRIPCIÓN: Verifica que un objeto tenga una propiedad específica,
             opcionalmente verificando también su valor.

EJEMPLO EN ESTE PROYECTO:
  expect(res.body).toHaveProperty('success');
  expect(res.body).toHaveProperty('data');
  expect(resBook).toHaveProperty('id');

USO: Para verificar la estructura de respuestas JSON sin importar el valor exacto.

----------------------------------------------------------------------
toBeDefined
----------------------------------------------------------------------
DESCRIPCIÓN: Verifica que el valor NO sea undefined.

EJEMPLO EN ESTE PROYECTO:
  expect(resAdmin).toBeDefined();
  expect(res.body.data).toBeDefined();

USO: Para confirmar que algo fue retornado, sin importar el contenido exacto.

----------------------------------------------------------------------
rejects.toThrow
----------------------------------------------------------------------
DESCRIPCIÓN: Verifica que una promesa rechazada lanza un error con
             el mensaje especificado.

EJEMPLO EN ESTE PROYECTO:
  await expect(bookAppointment(...)).rejects.toThrow('Mascota no encontrada');
  await expect(removeAppointment(...)).rejects.toThrow('Cita no encontrada');

USO: Fundamental para probar los caminos de error.

6. PRUEBAS ASÍNCRONAS
================================================================================

La mayoría del código de este microservicio es asíncrono (usa Promises y async/await)
porque realiza operaciones de red (base de datos, APIs externas).

----------------------------------------------------------------------
async/await
----------------------------------------------------------------------
DESCRIPCIÓN:
  Todas las pruebas usan "async" y "await" para esperar que las promesas
  se resuelvan o rechacen antes de verificar el resultado.

EJEMPLO:
  it('getAppointment exitoso', async () => {
    const result = await getAppointment('appt1', userAdmin, token);
    expect(result).toBeDefined();
  });

----------------------------------------------------------------------
resolves (promesas exitosas)
----------------------------------------------------------------------
DESCRIPCIÓN:
  Verifica que una promesa se resuelva con el valor esperado.

EJEMPLO:
  await expect(removeAppointment('appt1', userAdmin)).resolves.toBeUndefined();
  // Confirma que la función void terminó sin error

----------------------------------------------------------------------
rejects (promesas rechazadas)
----------------------------------------------------------------------
DESCRIPCIÓN:
  Verifica que una promesa sea rechazada con un error específico.
  Esencial para probar los bloques catch y las validaciones de seguridad.

EJEMPLO:
  await expect(bookAppointment(user, body, token))
    .rejects.toThrow('No autorizado para esta mascota');

----------------------------------------------------------------------
mockResolvedValue / mockRejectedValue
----------------------------------------------------------------------
DESCRIPCIÓN:
  Configura mocks para que devuelvan promesas resueltas o rechazadas.

EJEMPLO:
  (appointmentRepo.findAppointmentById as jest.Mock)
    .mockResolvedValue({ id: 'appt1', ... });   // ← simula éxito

  (appointmentRepo.findAppointmentById as jest.Mock)
    .mockRejectedValueOnce(new Error('DB Error'));   // ← simula fallo

11. CONCLUSIÓN
================================================================================

El testing es una práctica esencial en el desarrollo de software profesional.
Para un microservicio crítico como el de gestión de citas veterinarias, las
pruebas garantizan que:

  1. EL CÓDIGO FUNCIONA CORRECTAMENTE: Los casos exitosos confirman que
     el flujo principal (reservar, editar, cancelar citas) opera como se espera.

  2. LOS ERRORES SE MANEJAN BIEN: Las pruebas de error verifican que el sistema
     falla de forma controlada, retornando mensajes claros en lugar de crashear.

  3. LA SEGURIDAD SE MANTIENE: Las pruebas de autorización confirman que un
     usuario no puede acceder ni modificar datos de otro usuario o clínica.

  4. EL CÓDIGO NO REGRESA (Regresión): Si en el futuro se modifica alguna
     función, los tests detectarán inmediatamente si algo se rompió.

  5. CONFIANZA PARA DESPLEGAR: Un conjunto de tests que pasan da confianza
     al equipo para publicar nuevas versiones sin miedo a introducir bugs.

La cobertura alcanzada (>80% en statements y lines) es considerada una buena
práctica en la industria. El porcentaje restante corresponde principalmente
a código de infraestructura interna (notificaciones fire-and-forget) que
por su naturaleza asíncrona y privada es difícil de cubrir directamente,
pero que fue ejercitado indirectamente a través de las pruebas de bookAppointment.
